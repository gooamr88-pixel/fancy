P0 — Fix before any real event uses this

The QR "pass" shown right after RSVP is not a real ticket. Guests can be turned away at the door.

frontend/src/app/[slug]/rsvp/steps/StepSuccess.js:134 generates the celebratory pass's QR from qrData={partyId ? `fancy-rsvp:${slug}:${partyId}` : null} — a plain string encoded client-side via the qrcode package. The door scanner (backend/controllers/checkinController.js:20) calls tokenService.verifyQrTicket(token), which requires an HS256 JWT carrying purpose:'qr_ticket' (backend/services/tokenService.js:102). A plain string is not a JWT — jwt.verify throws and the scanner returns INVALID_TICKET. The real signed ticket only exists once it's emailed later (after seating is finalized), and no in-app screen — including frontend/src/app/ticket/[token]/page.js, the page reached via that real emailed link — ever renders the actual working QR image the backend already serves at GET /public/qr/:token.png. Verified independently by reading both the frontend generator and the backend token service; this is not a UX nitpick, it is a functional defect with real-world consequences at a real wedding.

Fix: stop labeling the success-screen pass "scan to check in" until a real signed ticket exists (swap in the honest copy already used elsewhere: "Your entrance QR will be emailed once seating is confirmed"), and render <img src=`${API_URL}/public/qr/${token}.png`> on ticket/[token]/page.js so the one page guests actually receive by email shows a working, scannable code.

01 — Scorecard

Where the platform stands today

Each score reflects severity-weighted findings against the product's own stated bar (Apple / Airbnb / Stripe / Linear), not against an average SaaS app. Bands: below 50, 50–69, 70+.

57/100
UX Score

The idempotent RSVP engine and single-scroll wizard show real craft, but the two highest-frequency mobile jobs — seat a guest, navigate the admin panel — are functionally broken, and the entrance pass doesn't match what the scanner accepts.

63/100
UI Score

Strong token foundation (brand palette, clamp() type scale) undercut by 11 button recipes and 18 card recipes fighting each other across the app.

46/100
Mobile Score

The guest-facing flow was clearly mobile-considered; the organizer dashboard and admin panel are desktop-shrunk, with entire pages carrying zero @media queries.

54/100
iPhone Score

viewport-fit=cover and bottom safe-area padding are correctly wired, but toast isn't safe-area-aware at the top, ~15 form surfaces still trigger Safari zoom, and there's no manifest/theme-color despite prepared icon assets.

59/100
Android Score

Fewer Android-specific defects surfaced, but the structural bugs — native HTML5 drag-and-drop, fixed CSS grids, hover-only tooltips — break identically on Chrome Android.

44/100
Accessibility Score

Multiple WCAG AA contrast failures on primary CTAs and status badges, three keyboard-unreachable click targets (one blocks the whole demo flow), and no modal has a real focus trap.

67/100
Performance Score

Good instincts — next/font, dynamic imports for qrcode/html5-qrcode/html-to-image, code-split wizard stages — offset by dead marketing components and triple-confetti at the one moment guests need to read their pass.

38/100
Design Consistency Score

The lowest score in the audit: two parallel token systems (marketing vs. admin), zero shared modal primitive across 5+ implementations, and measurably drifting spacing/radius/shadow values page to page.

56/100
Code Quality Score

Clean hook architecture (useAuth, useIdempotentRsvpSubmit) and a real token layer in admin/theme.js, offset by heavy inline-style sprawl, DOM-mutation anti-patterns, and orphaned dead code referencing Tailwind classes that don't exist.

Fix before launch

Grouped by product area. Each entry names the exact file/line, the concrete failure, and the specific fix — no generic advice.

Guest RSVP → Seating → QR Pass4 critical
CriticalQR passticket/[token]/page.js:121

The one page reached via a guest's real, emailed ticket link never renders the QR itself — only name, party size, and seating. Combined with the P0 above, no in-app surface ever shows a guest a functioning entrance code; their only fallback is digging through an email client.

CriticaliOS Safari / FormsGuestUI.js:773, rsvp/styles.js:4

Every field in the core RSVP form — name, email, meal choice, custom questions, notes — is styled at 14px, well under the 16px iOS Safari needs to avoid auto-zoom on focus. checkin/page.js:15 already sets 16px with the comment "prevents iOS focus-zoom" — the team knows the fix, it just never reached the guest form.

Fix — bump inputStyle/S.inputBase and every inline 14px input in StepPartyDetails.js/StepCustomQuestions.js to 16px.

CriticalFirst impressioncomponents/guest/DigitalEnvelope.js

The envelope-opening gate has no skip control and no memory across mounts. A guest returning to check their status, or navigating back from the seating map, is forced through the full tap-and-wait sequence again — on top of the separate envelope already shown on the event landing page moments earlier. GuestEnvelopeReveal.js:619 already implements a skip affordance elsewhere in the same codebase.

Fix — add the same persistent "Skip" control, and remember envelopeDismissed per-slug in sessionStorage.

CriticalQR passGuestPassGenerator.js:198

The entire pass card — front and back — is one onClick flip target. A nervous guest tapping/zooming their phone while approaching the scanner can flip the card away from the QR face at the exact moment they need to show it.

Fix — move the flip trigger to a small explicit "Details" affordance; keep the QR face the default, sticky state.

Organizer Dashboard8 critical
CriticalSeating mapdashboard/seating-map/page.js:191, 1255, 1348

The entire 1,786-line page is a fixed 3-column desktop grid with zero @media queries anywhere in the file. Worse: seating a guest is done by dragging their name onto a table using native HTML5 drag-and-drop (draggable="true", onDragStart/onDrop), which simply does not fire on touch without a polyfill; zoom is mouse-wheel only. An organizer at the venue on a phone cannot open a usable layout, let alone seat anyone — the single most severe mobile gap in the product.

Fix — rebuild the guest→table interaction on Pointer Events with a "tap guest, then tap table" touch mode; add pinch-to-zoom; collapse the 3-column grid into stacked/swipeable panels under 900px.

CriticalData tablesRSVPsTab.js:449, SeatingManager.js:130, page.js:1072

RSVPs and the seating-manager guest list both render as raw <table> with minWidth:720 wrapped only in overflowX:auto — the classic mobile anti-pattern — while GuestsTab.js next door already solved this with a card layout.

Fix — reuse GuestsTab's existing card pattern below 768px in both tables.

CriticalResponsive layoutGuestsTab.js:450

gridTemplateColumns:'repeat(2,1fr)' with no media query anywhere in the file — on a 375px screen each guest card is squeezed to ~170px, breaking the name/badge/meal-pill layout.

Fix — grid-template-columns:1fr under 640px, copying OverviewStatCards.js:340's existing breakpoint pattern.

CriticalModals / keyboardAddGuestModal.js:122, EditGuestModal.js:201

Neither modal sets maxHeight/overflowY on its panel. On a short viewport with the on-screen keyboard open, the Save/Cancel footer can be pushed off-screen with no way to scroll to it — a guest edit can become literally unfinishable.

Fix — match ImportGuestsModal.js:155's own pattern: maxHeight:'90vh', scrollable body, sticky footer.

CriticalFormsEventSettings.js (1,525 lines)

Zero @media queries in the file. Every 2-column row — dates/location/RSVP-privacy, and every event-type sub-form (wedding, corporate, engagement, birthday, gala) — stays 2 columns at any width, crushing inputs on a phone. This is the largest, most-used settings surface in the dashboard.

Fix — wrap the repeated row grid in a class with one @media (max-width:640px){grid-template-columns:1fr} rule, mirroring OrganizerProfile.js:761 which already solves this in the same dashboard.

CriticalFormsdashboard/campaigns/page.js:503, 507, 533

The SMS campaign composer is built on fixed grids (2fr 1fr, 1fr 1fr 1fr, 7fr 5fr) with zero @media queries — the message editor and audience panel are unusable side-by-side on a phone.

Fix — stack to one column under 900px; pair with a sticky mobile header (current back-affordance is a plain text link).

CriticalNavigationdashboard/page.js:506

Below 1024px, the sidebar becomes a hidden off-canvas drawer with no persistent way to jump between Events/Guests/RSVPs/Seating/Campaigns — every switch costs two taps, and the hamburger sits top-left, the worst spot for one-handed thumb reach.

Fix — add a persistent bottom tab bar (Overview / Guests / RSVPs / Seating / More) for the top 4-5 destinations.

CriticalDesign systemFormBuilder.js:213, InlineFormBuilder.js:239

Two near-identical "add a custom field" UIs in the same wizard get different mobile treatment — one collapses to 1 column at 768px, its sibling has no matching rule at all.

Super Admin Dashboard3 critical
CriticalNavigation_components/AdminShell.js:30, 93

useState(true) — the mobile nav drawer defaults open on every viewport, and onNavigate={() => {}} is a no-op, so tapping any nav link never closes it. Every phone visit to /admin/* fights a full-screen drawer + backdrop on first paint and after every navigation.

CriticalLayout gridsadmin/(panel)/config/page.js:391, 455, 467, 476, 549, 621

CriticalShared table_components/DataTable.js:38-133

Authentication (Login / Register / Forgot Password)5 critical
CriticalAccessibility / Contrastlogin/page.js:467, 414; register/page.js:637, 622

Every primary CTA and "gold link" (Forgot?, Create an account, Resend Code) uses raw #B8944F on white ≈ 2.84:1 — fails WCAG AA. globals.css:19 already defines a corrected --gold-cta:#8A6D34 (≈4.86:1, passes AA) used by the shared .btn-gold class — the auth pages bypass it and reintroduce the exact failure it was built to fix.

CriticalAccessibility / Contrastregister/page.js ~634, forgot-password/page.js:416

Password-requirement hint text at #99938A on white/#FAFAF8 ≈ 3.05:1 — fails AA.

Fix — darken to roughly #6B655C or darker.

CriticalValidation / A11ylogin/page.js:26, register/page.js:63, forgot-password/page.js:107

Every validation failure across all 3 pages surfaces only as a global toast — no input ever gets aria-invalid/aria-describedby or a visible error border, so nothing tells a sighted or screen-reader user which field is wrong.

CriticalForms / Autofillforgot-password/page.js:252-288

The reset form contains no input carrying the account's email inside the <form> — Chrome/Safari's "update saved password" heuristic expects an associated username field to correctly save the new credential.

Global Foundation & Shared Components4 critical
CriticalLayeringcomponents/Toast.js:70 vs AddGuestModal.js:117 and others

Toast and several dashboard modals share the exact same z-index:1000. Since modals mount later in the DOM, they paint on top — AddGuestModal.js:88 fires a toast while its own modal is still open, hiding the feedback the user most needs to see.

Fix — give the toast viewport a dedicated top layer, e.g. z-index:100001, above every modal.

CriticalAccessibilityLogoutModal.js, admin/_components/Modal.js, all dashboard modals

No shared modal primitive exists. None of the dashboard modals implement role="dialog", aria-modal, Escape-to-close, or a focus trap — keyboard/VoiceOver users can tab straight through the overlay into the page behind it.

Fix — extract one <Dialog> primitive (portal, ARIA roles, Tab-loop, Escape handler, body-scroll lock, focus restore) and migrate every one-off overlay onto it.

CriticalAccessibility / Formscomponents/PlacesAutocomplete.js:40, 104

Never accepts or forwards id/aria-invalid/aria-describedby, and its rendered input has no id. Call sites use a bare <label> with no htmlFor, so every venue-search field has no accessible name at all.

Fix — accept and spread id/aria-* props onto the input; wrap call sites in the existing FormField component.

CriticalError handlingmiddleware.ts:9, apiClient.js:37

Both the edge middleware and the API client's 401 handler silently redirect to /login on session expiry with zero explanation. authErrors.js already has the right copy ("Your session has expired…") but it's never actually invoked for this path — dead code.

Fix — append &reason=expired on redirect and toast the existing message on the login page when present.

06 — What's already right

Preserve these — don't refactor away real strengths

Genuine engineering and design strengths found during the audit

  • Idempotent RSVP engine (useIdempotentRsvpSubmit/useRsvpResolver) — a well-designed, race-safe single source of truth for the RSVP lock/edit/reconciliation lifecycle.
  • Single-scroll RSVP redesign deliberately avoids classic step-wizard back-button bugs; browser back behaves correctly.
  • Prior mobile/a11y remediation pass already in globals.css (tagged MOB-*/A11Y-*) correctly ships viewport-fit=cover, bottom safe-area padding on floating CTAs, a global :focus-visible ring, and prefers-reduced-motion handling.
  • Create-event wizard's Stage 1 template picker ships a genuinely separate, well-built mobile carousel layout — proof the team can execute mobile-first when they choose to.
  • Payment handling correctly delegates all PCI-sensitive card entry to Stripe Checkout rather than a custom form — the right call, already made.
  • Upgrade/paywall UX (UpgradeModal/FeatureGate) is a strong, non-dead-end pattern: clear feature list, distinct CTA copy, dismissible.
  • Onboarding empty states across Events, RSVP trend, upcoming events, and activity feed all ship purpose-built illustrated states with helpful CTAs, not blank tables.
  • Performance instincts: next/font, dynamic imports for qrcode/html5-qrcode/html-to-image only where used, and code-split wizard stages via next/dynamic.
  • Custom OTP/backend auth token service (tokenService.js) is deliberately unified with an explicit purpose discriminator specifically to close a prior cross-token-type vulnerability — good defensive design, documented in its own code comments.

09 — Production Readiness

Final assessment

Not production-ready — conditional on Phase 0

Ship-blocked by one functional defect, not a design problem.

The marketing site, authentication, and guest RSVP experience are close: the visual language, typography scale, and core interaction models (single-scroll RSVP, idempotent submit engine) already clear a premium bar. What blocks production is narrower and more fixable than the finding count suggests. One functional defect — the mismatched QR ticket — must be fixed before a single real event runs through this platform; it is a data-integrity bug wearing UX clothing, not a polish item. Two entire admin-facing surfaces (the super-admin panel and the organizer's seating map) are effectively unusable on a phone today, which matters because the platform's own pitch — "the organizer running around a venue on their phone" — depends on exactly the page that currently doesn't work on one. Once Phase 0 lands, this platform is a realistic candidate for the Apple/Airbnb/Stripe bar it's aiming for; today it is a strong desktop product with a mobile-considered guest funnel bolted alongside a mobile-shrunk operator experience.
